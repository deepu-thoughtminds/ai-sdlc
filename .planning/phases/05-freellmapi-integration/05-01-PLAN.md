<task type="auto">
  <name>Task 2: Update docker-compose.yml — replace Ollama stub with real freellmapi service</name>
  <read_first>docker-compose.yml</read_first>
  <files>docker-compose.yml</files>
  <action>
Replace the existing freellmapi service block in docker-compose.yml. The current service
uses an Ollama image on port 11434 with the FREELLMAPI_MODELS env var. Replace it entirely.

Current freellmapi service (lines 1–16 of docker-compose.yml):
  freellmapi:
    build:
      context: ./freellmapi
      dockerfile: Dockerfile
    restart: unless-stopped
    environment:
      - FREELLMAPI_MODELS=${FREELLMAPI_MODELS}
    networks:
      - ai-sdlc-net

Replace with:
  freellmapi:
    build:
      context: ./freellmapi
      dockerfile: Dockerfile
    restart: unless-stopped
    ports:
      - "3001:3001"
    environment:
      - ENCRYPTION_KEY=${FREELLMAPI_ENCRYPTION_KEY}
      - PORT=3001
      - HOST_BIND=0.0.0.0
    volumes:
      - freellmapi_data:/app/data
    networks:
      - ai-sdlc-net

Key changes explained:
- `ports: "3001:3001"` — exposes port 3001 to host (was no port mapping before; the
  Ollama stub was on 11434). This allows the freellmapi dashboard to be reached at
  http://localhost:3001 from the host machine.
- `ENCRYPTION_KEY` — freellmapi uses this env var name (not FREELLMAPI_ENCRYPTION_KEY)
  internally for its own key store encryption; we supply the value from our env var.
- `PORT=3001` and `HOST_BIND=0.0.0.0` — required for the Node.js process inside the
  container to bind on all interfaces so the backend hermes service can reach it by
  container name.
- `volumes: freellmapi_data:/app/data` — mounts the named volume at /app/data where
  freellmapi persists its encrypted key store SQLite file. Adjust the mount path if
  inspection of the freellmapi repo reveals a different data directory (check for
  `DATA_DIR` or similar in the freellmapi source).
- Remove `FREELLMAPI_MODELS` env var — no longer used (freellmapi uses its own model
  routing via "auto" alias in the LLM router).

Also add the `freellmapi_data` named volume to the top-level `volumes:` block:
  volumes:
    app_data:
    freellmapi_data:

If the executor finds from Task 1 that the freellmapi repo does NOT have a root-level
Dockerfile, update `dockerfile: Dockerfile` to `dockerfile: ../Dockerfile.override`
(using the override created in Task 1).

Also verify the `hermes` service `depends_on` entry still references `freellmapi` — it
should already, but confirm it is present. No change needed if it's there.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host → freellmapi container | HTTP requests to port 3001 from host for dashboard access and smoke tests |
| hermes/backend → freellmapi | Inter-container calls on ai-sdlc-net; Bearer token required |
| freellmapi → provider APIs | Outbound HTTPS from freellmapi container to Gemini/OpenRouter/Groq; key stored encrypted in freellmapi_data volume |
| .env file → container env | FREELLMAPI_ENCRYPTION_KEY and FREELLMAPI_API_KEY loaded at compose runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | FREELLMAPI_ENCRYPTION_KEY in .env | mitigate | .env.example uses placeholder; .env is in .gitignore (verify); document that key must be generated fresh per deployment |
| T-05-02 | Information Disclosure | FREELLMAPI_API_KEY in .env | mitigate | Same .gitignore protection; key generated via dashboard after first run, never stored in source |
| T-05-03 | Tampering | freellmapi_data Docker volume | accept | Volume is local Docker-managed storage; no network exposure; Docker daemon controls access |
| T-05-04 | Elevation of Privilege | freellmapi dashboard accessible on host port 3001 | mitigate | Document in FREELLMAPI_SETUP.md that port 3001 should NOT be exposed on production hosts; use firewall rules in non-dev environments |
| T-05-05 | Spoofing | npm package install during `docker compose build` of freellmapi | mitigate | freellmapi is a git submodule pinned to a specific commit SHA — executor must verify `git submodule status` shows clean SHA, not a floating branch ref |
| T-05-SC | Tampering | npm/pip/cargo installs (freellmapi Dockerfile) | mitigate | Submodule is pinned to a commit SHA; review Dockerfile after Task 1 clone to confirm no `curl | bash` or unversioned install steps |
</threat_model>
