|----------|-------------|
| Jira webhook → webhook router | Untrusted Jira payload enters; webhook secret validates origin |
| webhook router → hermes help-comment | Jarvis posts a help comment back to the originating Jira issue — only valid commands are listed, no user input echoed back |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-05 | Information Disclosure | webhook.py help-comment jira_token | mitigate | Decrypted token used for hermes_post_comment but never logged; only issue_key logged at INFO |
| T-14-06 | Spoofing | Agent-comment guard in webhook | mitigate | AGENT_BODY_MARKER check still fires before parse_mention — jarvis won't respond to its own help comments |
| T-14-07 | Denial of Service | Help-comment posting on every unknown intent | mitigate | Agent-comment guard prevents recursive loop (jarvis won't respond to its own comment); human can trigger at most one help comment per genuine @jarvis mention |
| T-14-08 | Elevation of Privilege | start_coding / merge_pr stub handlers | accept | Stubs return {"routed_to": "pending_phase_N"} and do nothing — no privileged action taken until Phases 16-17 wire real pipelines |
| T-14-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan |
