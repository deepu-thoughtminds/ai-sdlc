Build `backend/services/dev_pipeline.py` — the orchestrator that wires comment-history search, Confluence fetch, repo clone, code generation, and PR creation into a single `run()` coroutine — and replace the `start_coding` stub in `backend/routers/webhook.py` with real PipelineState-backed background task scheduling, following the exact idempotency-guard pattern already used by the `architecture` branch.

This plan connects all of it end-to-end and wires it to the webhook so `@jarvis start coding` actually triggers it.

Output: `backend/services/dev_pipeline.py` (new orchestrator), `backend/routers/webhook.py` (start_coding branch rewritten), plus tests for both.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Jira comment → webhook → dev_pipeline.run() | Untrusted Jira comment triggers an autonomous code-generation and PR-creation pipeline |
| dev_pipeline.run() → freellmapi | Jira story description + Confluence architecture content + codebase context interpolated into an LLM prompt |
| dev_pipeline.run() → GitHub | Decrypted github_token used to clone, push, and open a PR against the configured repo |
| dev_pipeline.run() → filesystem | Generated FileChange.content written to a temp workspace directory then committed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-05 | Information Disclosure | dev_pipeline.run() prompt construction | mitigate | Only issue_key/summary/description/architecture_content/codebase_context interpolated into generate_code_changes prompt — no jira_token/github_token/confluence_token ever passed to code_generator (carries forward T-06-01 from Phase 15) |
| T-16-06 | Information Disclosure | dev_pipeline.run() logging | mitigate | Only issue_key/github_repo slug logged; jira_token/github_token/confluence_token never logged (carries forward T-05-01/T-07-01/T-09-01 conventions) |
| T-16-07 | Denial of Service | dev_pipeline.run() workspace cleanup | mitigate | try/finally around clone→codegen→PR block guarantees shutil.rmtree(workspace_path, ignore_errors=True) runs even on exception, preventing temp-directory accumulation on repeated failed runs |
| T-16-08 | Repudiation | dev_pipeline.run() failure path | mitigate | WR-03 convention: failure always attempts a Jira comment notification before returning, so failures are visible in ticket history without server-log access |
| T-16-09 | Tampering | webhook.py start_coding idempotency guard | mitigate | PipelineState(status="running") created and committed BEFORE asyncio.create_task — prevents duplicate concurrent dev pipelines for the same ticket from racing to push conflicting branches/PRs |
| T-16-10 | Elevation of Privilege | generate_code_changes / apply_commit_push_and_open_pr autonomous write access | accept | v1.5 scope explicitly grants the agent autonomous code-change + PR-creation rights (per project CLAUDE.md "Dev stage is fully autonomous code changes"); PR review remains a human gate before merge (PR is opened, not auto-merged — PRMERGE-01/02 in Phase 17 require an explicit separate `@jarvis merge pr` trigger) |
</threat_model>
