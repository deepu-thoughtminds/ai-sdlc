<objective>
Build the two reusable primitives PRMERGE-01/02 depend on, before wiring the orchestrator:

1. `find_and_merge_pr()` in `pr_creator.py` — locates the open PR for a Jira issue by branch name (`jarvis/issue-{key}`) or title match, and merges it into `main` via the GitHub REST API.
2. A Jira status-update path — `HermesMCPClient.transition_issue()` (hermes container) exposed via a new `/jira/status` FastAPI endpoint, consumed by a new `backend/services/hermes_client.update_status()` async wrapper.

Purpose: Phase 17's orchestrator (Plan 02) needs both primitives as black-box building blocks. Splitting them into their own TDD plan keeps each RED→GREEN cycle focused and lets Plan 02 stay a thin orchestration layer.

Output: `find_and_merge_pr` (with `MergeResult` dataclass), `update_status` async wrapper, `transition_issue` MCP method, `/jira/status` hermes endpoint — all unit-tested in isolation.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| backend -> GitHub REST API | github_token (decrypted) sent over HTTPS to api.github.com; merge action is destructive (modifies main branch history) |
| backend -> hermes -> mcp-atlassian | jira_token (decrypted) sent over HTTP to hermes container, then to mcp-atlassian; status transition mutates Jira issue state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-17-01 | Information Disclosure | pr_creator.find_and_merge_pr | mitigate | github_token only ever placed in the `Authorization` header dict; never interpolated into f-strings, exception messages, or logger calls — verified by Task 1 Test 6 |
| T-17-02 | Information Disclosure | hermes_client.update_status / mcp_client.transition_issue | mitigate | jira_token only passed via JiraCredentials / payload dict; logger.warning in update_status logs only issue_key and exception text, never jira_token — verified by Task 2 Test 4 |
| T-17-03 | Repudiation / unhandled exception | mcp_client.transition_issue | mitigate | Wrapped in try/except returning False on any MCP tool-call failure (e.g. invalid transition name for the project's workflow) so a Jira status-transition failure never propagates into the merge_pipeline orchestrator's exception handler (Plan 02) |
| T-17-04 | Tampering | pr_creator merge endpoint (PUT .../merge) | accept | GitHub API enforces merge permissions server-side using the provided token's scopes; this codebase trusts the stored project token's authorization level (same trust model as existing apply_commit_push_and_open_pr) |
</threat_model>
