<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Jira webhook -> webhook.py | Untrusted comment body parsed into mention_result.action by upstream LLM intent classifier (Phase 14) — merge_pr is a destructive action triggered by free text |
| webhook.py background task -> merge_pipeline.run -> GitHub API | github_token decrypted and used to merge a PR into main — irreversible repository state change |
| merge_pipeline.run -> Jira (status transition + comment) | jira_token decrypted and used to mutate Jira issue state and post content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-17-05 | Information Disclosure | merge_pipeline.run | mitigate | github_token/jira_token decrypted locally and passed only to find_and_merge_pr/update_status as function args; never interpolated into comment_text or logger calls — verified by Task 1 Test 6 |
| T-17-06 | Tampering / Repudiation (duplicate merge race) | webhook.py merge_pr branch | mitigate | Idempotency guard mirrors architecture/start_coding pattern: PipelineState(status="running") created and committed BEFORE asyncio.create_task; only "running" status blocks a duplicate webhook from scheduling a second merge attempt — verified by Task 2 Test 2 |
| T-17-07 | Denial of Service (unhandled exception on no-PR-found) | merge_pipeline.run | mitigate | find_and_merge_pr returning None is treated as a normal (non-exceptional) control-flow branch; pipeline posts an informative comment and sets status="complete" rather than "failed" — verified by Task 1 Test 1 |
| T-17-08 | Tampering (session leakage across request/background boundary) | webhook.py _run_merge_background | mitigate | CR-02 — request-scoped db session is never captured by the background closure; only primitive project_id/issue_key/strings are captured, and a fresh SessionLocal() is opened inside the background coroutine — verified by Task 2 Test 3 |
| T-17-09 | Repudiation (Jira status update silently fails) | merge_pipeline.run update_status call | accept | update_status already returns False without raising (Plan 01, T-17-03); merge_pipeline does not retry or escalate a status-update failure beyond the existing comment confirming the merge SHA — acceptable because the merge itself (the irreversible action) already succeeded and is confirmed via the Jira comment regardless of status-transition outcome |
</threat_model>
