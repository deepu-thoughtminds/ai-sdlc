Purpose: Phase 18 built the scan service and the onboarding trigger; this plan adds the second trigger point (post-merge) so the snapshot stays current as code changes land, not just at project creation.
Output: `merge_pipeline.py` calls `codebase_scan_service.run()` after a successful merge, isolated from the merge pipeline's own success/failure state. New/updated tests in `test_merge_pipeline.py` prove the wiring and the isolation.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| merge_pipeline.run() -> codebase_scan_service.run() | Internal call, plaintext github_token/github_repo passed as function args (already decrypted in Step 2) |
| merge_pipeline.run() -> Jira comment body | Exception text from a failed re-scan must never leak into the user-visible Jira comment |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-19-01 | Information Disclosure | merge_pipeline.py re-scan try/except | mitigate | Re-scan exception is logged via `logger.warning` only (server-side); never interpolated into `comment_text` or any value passed to `hermes_post_comment` — the SHA confirmation comment built in Step 6 runs before the re-scan hook and is never modified by it |
| T-19-02 | Tampering | merge_pipeline.py state_row | accept | Re-scan failure must not alter state_row.status; enforced by keeping the re-scan try/except fully separate from the Step 7 `db.commit()` and the outer except block — no shared mutable state between them beyond reading already-decrypted github_repo/github_token |
| T-19-03 | Denial of Service | merge_pipeline.py control flow | mitigate | Re-scan call is wrapped in try/except so a hung/erroring GitHub API call (already bounded by codebase_scan_service's own 30s httpx timeout) cannot propagate an unhandled exception that would otherwise flip state_row.status to "failed" for a merge that actually succeeded |
</threat_model>
