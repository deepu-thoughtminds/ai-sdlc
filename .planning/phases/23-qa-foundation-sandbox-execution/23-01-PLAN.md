## Context

Phase 23 establishes the QA execution substrate before any test-running logic exists. This plan (Wave 1) delivers three independent infrastructure pieces that subsequent plans depend on:

1. **qa-sandbox Docker image** — an isolated container with both Python QA toolchains (ruff, mypy, bandit) and JS/TS toolchains (node, eslint, typescript/tsc, npm). The backend mounts a cloned workspace as a volume and invokes `docker run --rm -v {workspace}:/workspace qa-sandbox <tool> /workspace` — no Docker-in-Docker required.

2. **docker-compose.yml qa-sandbox service** — declares the image so `docker compose build` builds it; also mounts `/var/run/docker.sock` into the backend service so `test_executor.py` can issue `docker run` calls at runtime.

3. **PipelineState.qa_attempt column** — an Integer nullable column added to the ORM model and both Pydantic schemas. It is set to 0 when a QA run starts (before execution) and incremented before each fix attempt. `nullable=True` means `None` = not yet in QA; `0` = first attempt started. This matches the Phase 10 pattern (complexity column added with same nullable convention).

## Implementation

### Step 1 — qa-sandbox/Dockerfile

Create `qa-sandbox/Dockerfile`. Use `python:3.12-slim` as the base (matches backend). Install:
- Python tools via pip: `ruff`, `mypy`, `bandit`
- Node.js via apt (nodejs + npm from Debian bookworm-backports or nodesource)
- JS/TS tools via npm globally: `eslint`, `typescript` (provides `tsc`)

The image does not run a long-lived process — it is invoked with `docker run --rm` per command. Set the default CMD to `["bash"]` or leave it absent; the entrypoint is provided by the caller.

```
FROM python:3.12-slim

RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        git \
        curl \
        nodejs \
        npm \
    && rm -rf /var/lib/apt/lists/*

RUN pip install --no-cache-dir ruff mypy bandit

RUN npm install -g eslint typescript

WORKDIR /workspace
```

Key decisions:
- `WORKDIR /workspace` matches the mount point the backend will use (`-v {path}:/workspace`)
- No CMD — the backend supplies the full command via `docker run ... qa-sandbox ruff check /workspace`
- `git` is included because some static analysis tools inspect `.git` metadata

### Step 2 — docker-compose.yml changes

Make two scoped edits to `docker-compose.yml`:

**Edit A — add qa-sandbox service** (append before the `networks:` key):

```yaml
  qa-sandbox:
    build:
      context: ./qa-sandbox
      dockerfile: Dockerfile
    image: qa-sandbox
    networks:
      - ai-sdlc-net
```

Note: This service has no `command` or `ports` — it is never started as a long-lived service. It exists so `docker compose build` builds the image and tags it as `qa-sandbox`. The backend invokes it on-demand via `docker run --rm`.

**Edit B — add Docker socket volume and env var to the backend service**:

Under `backend.volumes`, add:
```yaml
      - /var/run/docker.sock:/var/run/docker.sock
```

Under `backend.environment`, add:
```yaml
      - DOCKER_SOCKET=/var/run/docker.sock
      - QA_SANDBOX_IMAGE=${QA_SANDBOX_IMAGE:-qa-sandbox}
```

The `QA_SANDBOX_IMAGE` env var allows overriding the image name in CI without rebuilding; defaults to `qa-sandbox`.
