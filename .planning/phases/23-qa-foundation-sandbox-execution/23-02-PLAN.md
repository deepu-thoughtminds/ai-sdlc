## Context

Wave 2 implements the two service modules that form the QA pipeline's execution core. The architectural boundary established here is: **the LLM generates test files; the orchestrator (qa_pipeline.py) decides when to run them; test_executor.py runs them safely**. Phase 23 only establishes the boundary and the static analysis path — LLM test generation comes in Phase 24.

**test_executor.py** is a pure-function module (no ORM, no async) — it reads the filesystem and shells out to Docker. This makes it easy to unit test with mocked subprocess calls.

**qa_pipeline.py** is the async orchestrator skeleton that mirrors `merge_pipeline.py`'s structure exactly: re-use or create a `PipelineState` row, try/finally cleanup, decrypt credentials, post Jira comment on outcome. Phase 24 will add LLM test generation inside this skeleton; Phase 25 will add the fix loop.

## Implementation

### Step 1 — backend/services/test_executor.py

Module docstring must reference: TESTEXEC-01, TESTEXEC-02, TESTGEN-02 and list threat mitigations T-23-01, T-23-02.

**Dataclasses:**

```
ToolchainCommand  — name: str, command: list[str]
    name: human-readable tool name (e.g. "ruff", "eslint")
    command: full docker run command list ready to pass to subprocess.run()

TestResult  — tool: str, returncode: int, stdout: str, stderr: str, timed_out: bool
    tool: same as ToolchainCommand.name
    returncode: process exit code (-1 if timed out)
    stdout/stderr: captured text output
    timed_out: True when subprocess.TimeoutExpired was caught
```

**detect_toolchain(workspace_path: str) -> list[ToolchainCommand]**

Logic (no LLM invoked):

1. Check for `pyproject.toml` or `setup.cfg` in `workspace_path`. If either exists, this is a Python project. Add these ToolchainCommands:
   - ruff: `["docker", "run", "--rm", "-v", f"{workspace_path}:/workspace", image, "ruff", "check", "/workspace"]`
   - mypy: `["docker", "run", "--rm", "-v", f"{workspace_path}:/workspace", image, "mypy", "/workspace", "--ignore-missing-imports"]`
   - bandit: `["docker", "run", "--rm", "-v", f"{workspace_path}:/workspace", image, "bandit", "-r", "/workspace", "-q"]`

2. Check for `package.json` in `workspace_path`. If present, this is a JS/TS project. Add:
   - eslint: `["docker", "run", "--rm", "-v", f"{workspace_path}:/workspace", image, "eslint", "/workspace", "--ext", ".js,.ts,.jsx,.tsx"]`
   - tsc: `["docker", "run", "--rm", "-v", f"{workspace_path}:/workspace", image, "tsc", "--noEmit", "--project", "/workspace/tsconfig.json"]` — only add if `tsconfig.json` exists in workspace_path
   - npm_audit: `["docker", "run", "--rm", "-v", f"{workspace_path}:/workspace", image, "npm", "audit", "--prefix", "/workspace"]`

3. Read the image name from `os.environ.get("QA_SANDBOX_IMAGE", "qa-sandbox")`.

4. Return the list (may be empty if workspace has neither Python nor JS/TS indicators — log a warning in that case).

Note: The command list is constructed purely from fixed strings and the workspace_path / image name from env — no user-supplied tool names are ever interpolated (T-23-01).

**run_command(cmd: ToolchainCommand, timeout: int = 120) -> TestResult**

```
try:
    result = subprocess.run(
        cmd.command,
        capture_output=True,
        text=True,
        timeout=timeout,
    )
    return TestResult(
        tool=cmd.name,
        returncode=result.returncode,
        stdout=result.stdout,
        stderr=result.stderr,
        timed_out=False,
    )
except subprocess.TimeoutExpired:
    logger.warning("Command %s timed out after %ds", cmd.name, timeout)
    return TestResult(
        tool=cmd.name,
        returncode=-1,
        stdout="",
        stderr=f"Command timed out after {timeout}s",
        timed_out=True,
    )
```

Never catch generic Exception here — let unexpected errors propagate to the caller (qa_pipeline.py handles them in its try/except).

**run_static_analysis(workspace_path: str, timeout: int = 120) -> list[TestResult]**

```
commands = detect_toolchain(workspace_path)
results = []
for cmd in commands:
    logger.info("Running static analysis: %s", cmd.name)
    result = run_command(cmd, timeout=timeout)
    results.append(result)
    logger.info("%s exit=%d timed_out=%s", cmd.name, result.returncode, result.timed_out)
return results
```

### Step 2 — backend/services/qa_pipeline.py

Mirror `merge_pipeline.py` structure exactly. Module docstring references TESTEXEC-01, TESTEXEC-02, AUTOFIX-04 and threat mitigations T-23-01 through T-23-05.

**Class QAPipeline** (or a module-level `async def run(...)` — use the class form to match Phase 24-26 extension points):

```
async def run(
    project: Project,
    issue_key: str,
    issue_summary: str,
    issue_description: str,
    db: Session,
) -> str:
```

**Step-by-step inside run():**

Step 1 — Re-use or create PipelineState row (stage="qa", status="running"). Same pattern as merge_pipeline.py: query for existing running row, create one if absent.

Step 2 — Set qa_attempt before any execution (T-23-04):
```python
state_row.qa_attempt = 0
db.commit()
```

Step 3 — Decrypt credentials (github_token, github_repo, jira_token). Use `decrypt_credential` from `services.crypto`.

Step 4 — Clone a fresh workspace using `clone_repository(github_repo, github_token)` from `services.repo_clone`. This is always a fresh clone — never the workspace from dev or merge pipeline (TESTEXEC-02).

Step 5 — Wrap execution in try/finally:
```python
cloned = None
try:
    cloned = clone_repository(github_repo, github_token)
    results = run_static_analysis(cloned.workspace_path)
    # Phase 24 will add LLM test generation here
    comment_text = _format_static_analysis_comment(results, issue_key)
    state_row.status = "complete"
    db.commit()
except Exception as exc:
    state_row.status = "failed"
    try:
        db.commit()
    except Exception:
        db.rollback()
    comment_text = f"QA pipeline failed for {issue_key}. Check server logs."
    logger.exception("QA pipeline failed for %s: %s", issue_key, exc)
finally:
    if cloned is not None:
        shutil.rmtree(cloned.workspace_path, ignore_errors=True)
```

Step 6 — Post Jira comment using `hermes_post_comment`. Wrap in its own try/except so comment failure does not mask pipeline failure.

**_format_static_analysis_comment(results, issue_key) -> str** (private helper):

Formats a human-readable summary of static analysis results. For each TestResult:
- If `returncode == 0`: "ruff: PASSED"
- If `timed_out`: "mypy: TIMED OUT (120s)"
- Otherwise: "bandit: FAILED (exit 1)\n{stderr[:500]}"

Returns a string suitable for posting as a Jira comment. Truncates per-tool output to 500 chars to avoid enormous comments (Phase 24 will add structured reporting via QAREP-01).
