## Objective

Deliver a new, self-contained backend service `backend/services/app_container.py` exposing a single public context manager `managed_app_container(...)` that:

- **SERVE-01** — detects the serve command from the cloned repo's `package.json` (`preview` > `start` > `dev`), logs the choice, raises `ValueError` when none exist.
- **SERVE-02** — builds and serves the app in an ephemeral Docker container attached to the **caller-supplied** compose network, with a dynamically allocated host port (`-p 0:PORT`).
- **SERVE-03** — polls `GET /` against the container until HTTP 200, or raises `ContainerStartError` after a configurable timeout (default 60s).
- **SERVE-04** — always removes the container (`docker rm -f`) in a `finally` block — on success, failure, timeout, or unexpected exception.

This phase is **purely additive**: one new module + one new test file. No existing files are modified and no DB schema changes are introduced. The `qa_pipeline.py` caller integration (passing the live URL into the Playwright generator) is **Phase 28** (PWGEN-01/02), not this phase.

**Output:** `backend/services/app_container.py`, `backend/tests/test_app_container.py`.

**URL reachability (load-bearing design decision):** containers on `ai-sdlc-net` reach each other by **container name + container port** via Docker's embedded DNS — this is exactly how the existing Playwright runs reach `http://frontend:3000`. Therefore `managed_app_container` yields `http://<container_name>:<container_port>` (network-internal), NOT `http://localhost:<host_port>`. `localhost:<host_port>` only resolves from the host, not from the backend container (which runs the health-check) nor from the Phase 28 Playwright container (which consumes the URL). The dynamic host port (`-p 0:PORT`) is still published to satisfy SERVE-02 and aid host-side debugging, but it is not part of the returned URL.

Resolve the three optional params from their env defaults when `None`. Generate `name = f"jarvis-app-{uuid.uuid4().hex[:8]}"` (DNS-safe, no user input). ## threat_model

**Trust boundary:** the cloned **target repo is untrusted** — its `package.json` scripts and dependencies execute arbitrary code during `npm ci` / `npm run build` / serve, inside a container that joins the shared `ai-sdlc-net`.

| Threat ID | Category (STRIDE) | Component | Disposition | Mitigation |
|-----------|-------------------|-----------|-------------|-----------|
| T-27-01 | Tampering / Elevation | `npm ci`/`build`/serve of untrusted repo code in the started container | mitigate | Runs in an ephemeral, unprivileged container (no `--privileged`, no docker.sock mounted into the served container); torn down after use. |
| T-27-02 | Denial of Service | build/serve hangs or never becomes healthy | mitigate | `_wait_until_healthy` deadline (default 60s via `APP_CONTAINER_HEALTH_TIMEOUT`) raises `ContainerStartError`; teardown follows. |
| T-27-03 | Denial of Service / resource leak | orphaned containers accumulate across runs | mitigate | `docker rm -f` in `finally` (SERVE-04) plus `--rm` on `docker run` for the process-killed-mid-run case. |
| T-27-04 | Information Disclosure / lateral movement | served container can reach internal services (freellmapi, litellm, backend) on `ai-sdlc-net` | accept | Matches the existing Playwright-container pattern (also joins `ai-sdlc-net`); container is ephemeral and receives **no credentials** in its env. Rationale documented; network isolation deferred (future requirement). |
| T-27-05 | Injection | `workspace_path` / container name interpolated into the docker command | mitigate | List-form argv only (no shell); container name derived from `uuid4().hex` (no external input); `workspace_path` is an internally-generated `/tmp/jarvis-*` path. |
| T-27-SC | Tampering (supply chain) | new package installs in this phase | accept | None added — `httpx` already pinned in `requirements.txt`; no install task, no `## Package Legitimacy Audit` required. (The untrusted target-repo `npm ci` is covered by T-27-01's container isolation.) |

**Schema gate:** no DB schema changes are introduced in this phase — no migration task required.
